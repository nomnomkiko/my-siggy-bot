Key innovations include enshrined on-chain AI models, scheduled transactions (no external keepers needed), a model marketplace, native account abstraction, modular computational integrity, and a comprehensive agent framework. Ritual also serves other blockchains through its Infernet oracle network, which connects 8,000+ compute nodes.

The goal is to decentralize access to AI and expand what's possible on-chain.


WHY DOES THE WORLD NEED ANOTHER BLOCKCHAIN?
    Traditional blockchains were designed for simple, homogeneous transactions, every node executes the same basic operations. But modern applications demand complex computations like AI inference, ZK proofs, and TEE execution that existing chains can't efficiently handle.
    The core problems with existing chains:
        Limited compute: They only support operations the weakest node can run
        Wasteful re-execution: Every node redundantly re-executes every transaction
        No specialization: All nodes are treated the same regardless of hardware
        Inefficient pricing: Unique resources are priced identically
    Behind every magnitude increase in blockchain users lies an innovation that enables net-new behavior for the masses. In 2009, Bitcoin enabled anyone, anywhere to send and receive money without relying on centralized intermediaries. In 2015, Ethereum introduced smart contracts, enabling developers to build permissionless, composable financial applications.
    Yet, somewhere along the way, we lost sight of building for the net-new user, of expanding the status quo of what could be built with truly trustless and expansive computation. Today’s blockchains prioritize latency and throughput, optimizing for existing users rather than innovating. While these performance improvements matter, they merely scale current use cases instead of unlocking transformative new ones.
    Ritual was born from a single focus: enrich what users can do on-chain today to attract the users of tomorrow.
    The expedition began at the intersection of Crypto and Artificial Intelligence. Two things are simultaneously true: it has never been easier for developers to build with AI, crafting dynamic and personalized experiences, and it has never been more important to ensure that the AI technologies we adopt remain open, credibly neutral, and censorship resistant. To that extent, Ritual makes smart contracts actually smart, allowing users to natively tap into on-chain AI, backed by the same trustless properties of modern blockchains.
    However, Ritual isn’t just focused on Crypto x AI. By optimizing our underlying architecture for any type of heterogeneous compute, Ritual hyperscales the expressive computation that can be performed on a blockchain. Ritual is the best venue not just for developers looking to harness AI, ZK, or TEEs in their applications, but also for those experimenting at the edges of the unknown, defining new ways to consume underlying infrastructure.
    Our architecture efficiently prices, orchestrates, and exposes all types of compute, current or future, enabling Ritual to become the schelling point for all of web3.
    Not only will end-users consume applications natively through Ritual, but other blockchains themselves symbiotically benefit through Ritual. High-performance chains that require a prover network (MegaETH, Movement, etc.), modern Rollup-as-a-Service offerings that require efficient execution of heterogeneous compute (Conduit, Caldera, etc.), and the totality of IP, monetized via a verifiable marketplace (Story, Sentient, etc.) will all use Ritual.

What is the Ritual architecture?
    - EVM++
        EVM++ is a backwards-compatible extension of the Ethereum Virtual Machine (EVM) introduced by Ritual Chain, designed to enable developers to build "actually smart contracts" while using the same tooling they already know. It enhances the standard EVM with four major components:
            Scheduled Transactions: allow recurring, conditional invocation of smart contract functions at the top of a block, eliminating the need for external keeper bots (like Gelato or Chainlink Automation). This gives developers guaranteed top-of-block priority execution, predictable timing, dedicated fee markets, and conditional logic support — enabling truly autonomous on-chain applications like smart agents.
            Account Abstraction: is supported natively via EIP-7702, making Ritual one of the earliest chains to adopt this standard proposed by Vitalik Buterin. It introduces a new SetCodeTx transaction type that lets EOA accounts delegate a smart contract as their implementation, unlocking features like batch calls, gas fee sponsorship, arbitrary signing keys (P256, BLS), and advanced access control lists.
            EIP Extensions: represent Ritual's commitment to rapidly integrating in-demand Ethereum Improvement Proposals. Current extensions include: EIP-665 (Ed25519 signature verification), EIP-5027 (removal of the 24,576-byte contract code size limit with dynamic gas metering), EIP-5920 (a new PAY opcode to send ether without transferring execution context, preventing reentrancy attacks), and EIP-7212 (secp256r1/P-256 signature verification enabling Passkeys and hardware-based signing).
            Expressive Compute: provides heterogeneous compute precompiles that power entirely new application categories. Combined, these components allow developers to progressively adopt advanced Ritual Chain functionality while remaining fully compatible with existing EVM tooling.

    - Execution Sidecars
        Execution Sidecars are optimized components within the Ritual architecture designed to support expressive, heterogeneous compute—such as AI inference, zero-knowledge (ZK) proof verification, and Trusted Execution Environment (TEE) code execution
        They function as a specialized layer that allows the blockchain to handle complex tasks that go beyond standard smart contract capabilities
        These sidecars run and scale parallel to the Ritual execution client, invoked through abstraction via the EVM precompile interface. By default, these sidecars are purpose-built for execution and proof generation avoiding blockchain-specific overhead, run as an isolated process or remote to an execution client, and can be selectively enabled by node operators based on their available infrastructure, underlying hardware, or firewalled preferences.
        Because of their generic architecture, sidecars enable Ritual to easily service heterogeneous compute across any supported execution environment by connecting to their precompile-like interfaces (EVM precompiles, SVM precompiled programs, MoveVM system modules, etc.).
        Resonance is a state-of-the-art transaction fee mechanism designed to efficiently match supply and demand, providing users with optimally priced transaction execution

    - Resonance 
        Resonance is Ritual Chain's state-of-the-art transaction fee mechanism designed to efficiently match supply and demand in a world of increasingly heterogeneous blockchain transactions, offering users optimally priced transaction execution.
        The Problem: Blockchain transactions have grown increasingly complex — from Bitcoin's simple transfers, to Ethereum's smart contracts, to blobs (EIP-4844), to parallel execution on chains like Solana and Monad. Existing fee mechanisms like first-price auctions and multi-dimensional EIP-1559 are insufficient for this heterogeneity. They suffer from poor welfare guarantees as dimensions increase, require hard-coding resources into the protocol (updatable only via hard forks), and face challenges in setting correct base fees. For users, this means increased costs, limited support for new transaction types, pricing complexity, and failed executions.
        For Users, Resonance allows each transaction to specify a valuation for execution. Users can prioritize speed or cost, execute any type of compute without protocol-level constraints, and even execute transactions privately via multi-party assignment.
        For Nodes, operators specify a granular cost function for each transaction, can specialize in specific compute types (e.g., AI inference), execute any type of compute, and process unconflicting transactions in parallel.
        For the Network, transactions are efficiently matched optimizing for both cost and speed, node capacity is optimally allocated, and combined with Symphony, parts of the network can be specialized. The mechanism ensures all parties benefit from participating truthfully.
        Brokers are the key innovation under the hood — sophisticated, profit-seeking agents (analogous to block builders or MEV searchers) that compute matchings between transactions and nodes, pocketing the spread between transaction valuation and node execution cost. The Resonance mechanism ensures brokers behave predictably and fairly, offering reliability that Ethereum's current block-building process doesn't fully guarantee.
        Future work includes native pricing for scheduled transactions, decentralizing the auctioneer so anyone can run it, increased TPS for heterogeneous transactions, and research into posted-price mechanisms for an "Uber-like" pricing experience.

    - Symphony
        Symphony is Ritual Chain's novel consensus protocol designed to reduce replicated execution and enable high-performance, resource-intensive workloads. It builds upon an Execute-Once-Verify-Many-Times (EOVMT) model, fundamentally rethinking how blockchains process transactions.
        The Problem: Most blockchains today rely on replicated execution, where every node re-executes every transaction to verify correctness and apply state updates. While this works for simple, homogeneous computations (basic arithmetic, hash functions), it becomes redundant, inefficient, and unscalable for Ritual's heterogeneous workloads — such as AI inference opcodes that can take minutes or even hours to execute. Synchronously processing these in a replicated fashion would drastically increase block times and potentially halt block production entirely.
        Execute Once: Symphony solves this by having select compute nodes act as sole executors of workloads. Inference SPC opcodes are executed asynchronously off-chain by designated compute nodes in a specialized environment, generating succinct sub-proofs for the computation. Since execution and proof generation happen off-chain, they don't impact the on-chain critical path. Ritual prioritizes prover-friendly systems with linear or near-linear proof generation time, further optimizable through parallel multi-core or multi-machine setups.
        Verify Many Times: Rather than all validators verifying all proofs (replicated verification), Symphony employs a four-pronged scalability strategy: (1) Model partitioning — large models are split into smaller sub-models with separate sub-proofs; (2) Distributed verification — sub-proofs are assigned to different validator subsets via uniform deterministic assignment; (3) Verification committees — validators specialize in specific model types; and (4) Optimistic verification — blocks can be processed without waiting for full verification. Proof shards are stored off-chain, with blocks containing only outputs and proof references.
        Flexible Proof Systems: Symphony supports multiple proof systems — TEE Attestations for rapid hardware-backed verification, Zero-Knowledge Proofs for stronger cryptographic and privacy guarantees, and Hybrid Solutions combining both for fast confirmation with added security.
        Safety and Liveness: Symphony guarantees liveness through economic incentives (similar to Ethereum 2.0's reward/penalty model) and achieves probabilistic safety — with at least ⅔ of staked value controlled by honest participants, the chance of a safety violation is negligible, comparable to a secure hash collision.
        Key Benefits: Consensus becomes optimized (single-node execution), decentralized (distributed proof verification), and streamlined (collective validity consensus), eliminating inefficiencies while natively supporting node specialization.

    - Scheduled Transactions
        Scheduled Transactions are a native Ritual Chain capability that enables recurring, conditional invocation of smart contract functions at the top of a block, completely eliminating the need for external keeper bots like Gelato or Chainlink Automation.
        Homogeneous Compute: Ritual introduces a new predeploy contract called Scheduler, callable only by other smart contracts (including EIP-7702 delegated addresses). This contract maintains the state of ScheduledCall(s) and escrows future transaction fees. Developers set up invocations via Scheduler.schedule(), specifying function details, calldata, invocation frequency, an optional conditional execution function (returning True/False), and fee limit preferences. Cancellation and refunds are handled through Scheduler.cancel(). Block producers maintain the latest state of ScheduledCall(s) from the Scheduler. During intervals when a ScheduledCall should execute, nodes include a SystemTx transaction (akin to L2 networks like Optimism) that invokes the developer's smart contract at the top of their produced block, conditional on transaction validity and sufficient fees. To the developer's contract, this transaction behaves like any other invocation, except with a msg.sender equal to a constant SystemTxAccount.
        Heterogeneous Compute: Scheduled transactions containing heterogeneous compute use the same standard Scheduler interface but are executed via the Resonance fee mechanism. Each invocation of a ScheduledCall is priced just-in-time as if it were an independent, unscheduled transaction. Brokers receive a one-block heads-up to upcoming scheduled transactions for future allocations. These transactions are executed by assigned network subsets, with outputs and proofs gossiped to block producers for inclusion.
        Top-of-Block Inclusion: Scheduled transactions enjoy privileged top-of-block inclusion, enabling priority execution for critical automated operations like oracle updates and rebalancing mechanisms, with separately priced fee markets. This also powers enshrined oracles on Ritual Chain.
        Looking Forward: Ritual plans to extend Resonance to natively price scheduled transactions within the mechanism itself — reducing end-to-end finality time and enabling more robust pricing by treating ScheduledCall invocations as continual, predictable, yet cancellable future execution rather than disjoint events. Additionally, Ritual aims to embrace true PBS++ (proposer-builder separation) by enabling brokers to account for top-of-block preferential treatment during block construction.

    - Infernet ↔︎ Chain
        Infernet is Ritual's decentralized oracle network purpose-built for AI and heterogeneous compute workloads, launched in November 2023 as the first of its kind. It was designed as a lightweight framework powered by a network of Infernet nodes executing arbitrary workload containers. Today, there are over 8,000+ independent Infernet nodes with diverse hardware profiles and capabilities, ready to execute requests — all without any expectation of incentives.
        Core Capabilities: With its 1.0.0 upgrade, Infernet introduced highly requested features including audited on-chain payments, frameworks for verification of compute, and support for streaming responses. It provides secure model execution, result verification, and cross-chain integration capabilities, enabling developers to bring verified AI computation on-chain across any EVM-compatible blockchain.
        Dual Role in the Ecosystem: Infernet serves two complementary purposes. First, it operates as a standalone oracle network enabling any existing blockchain to access verified AI computation. Second, it is a core component of Ritual Chain's architecture, natively integrated to enable specialized compute delegation across the network. This dual role creates a comprehensive ecosystem for on-chain compute.

    6. Resonance
            Resonance is Ritual Chain's state-of-the-art transaction fee mechanism designed to efficiently match supply and demand in a two-sided market setting, offering users optimally priced transaction execution.
            The Problem — Heterogeneous Blockchains: Blockchain transactions have become increasingly complex and diverse over time. Bitcoin supported only simple arithmetic operations like transfers; Ethereum introduced smart contracts for arbitrary logic; blobs (EIP-4844) enabled data storage transactions; and chains like Solana, Monad, and MegaETH introduced parallel execution. As new applications like gaming and AI bring more complex computation on-chain, transaction heterogeneity will grow exponentially. The leading fee mechanism today — multi-dimensional EIP-1559 — is insufficient. As shown in Ritual's formal paper, it suffers from poor welfare guarantees as dimensions increase, requires hard-coding resources into the protocol (updatable only via hard forks), and faces information-theoretic challenges in setting correct base fees. For users, this translates to increased costs, limited support for new transaction types, pricing complexity, and failed executions.
            Resonance for Users: Each transaction specifies a valuation for execution. Users can prioritize speed or cost, execute any type of compute without protocol-level constraints, and even execute transactions privately via multi-party assignment.
            Resonance for Nodes: Operators specify a granular cost function for each transaction, can specialize in specific compute types (e.g., AI inference), execute any type of compute without constraints, and process unconflicting transactions in parallel.
            Resonance for the Network: Transactions are efficiently matched optimizing for both cost and speed, node capacity is optimally allocated, users retain full preference over execution, and combined with Symphony, parts of the network can be specialized. The mechanism ensures all parties benefit from participating truthfully.
            Brokers: The key innovation powering Resonance is the introduction of brokers — sophisticated, profit-seeking agents that compute matchings between transactions and nodes. Analogous to block builders or MEV searchers, brokers use informed priors to efficiently match transactions, pocketing the spread between transaction valuation and node execution cost. The Resonance mechanism's properties ensure brokers behave predictably and fairly — a level of reliability that Ethereum's current block-building process doesn't fully guarantee.
            Future Work: Ritual plans to extend Resonance in several areas: natively pricing scheduled transactions for economically efficient on-chain agents, decentralizing the auctioneer so anyone can run it, increasing TPS for heterogeneous transactions via streaming settings, and researching posted-price mechanisms for an "Uber-like" pricing experience.

    7. Symphony
        Symphony is Ritual Chain's novel consensus protocol designed to reduce replicated execution and enable high-performance, resource-intensive workloads. It builds upon an Execute-Once-Verify-Many-Times (EOVMT) model, fundamentally rethinking how blockchains process transactions.
            Existing Consensus Protocols — The Problem: Most blockchains today operate on replicated execution, where every node re-executes each block to verify transactions and compute state updates. While this works for simple, homogeneous computations (basic arithmetic, hash functions), Ritual's heterogeneous, resource-intensive workloads — such as AI inference opcodes that can take minutes or even hours — make replicated execution redundant, inefficient, and unscalable. Synchronously processing these would drastically increase block times and potentially halt block production entirely.
            Execute Once: Symphony adopts an Execute-Once model where inference SPC opcodes are executed asynchronously off-chain by a single designated compute node in a specialized environment. To eliminate the need for replicated re-execution, the executor generates succinct sub-proofs that validators can efficiently verify. Since execution and proof generation happen off-chain, they don't impact the on-chain critical path. Ritual prioritizes prover-friendly systems with linear or near-linear proof generation time, further optimizable through parallel multi-core or multi-machine setups.
            Verify Many Times: While Execute Once solves replicated execution, verifying proofs for large model inferences poses its own scalability challenges — proofs grow polylogarithmically with model complexity and can reach megabytes for models with tens of billions of parameters. Ritual employs a four-pronged scalability strategy: (1) Model partitioning — large models are split into smaller sub-models with separate sub-proofs; (2) Distributed verification — sub-proofs are assigned to different validator subsets via uniform deterministic assignment; (3) Verification committees — validators specialize in specific model types; and (4) Optimistic verification — blocks can be processed without waiting for full verification.
            Partitioned Proofs: The page details further optimizations including model partitioning into smaller sub-models with separate sub-proofs, proof sharding that breaks computation traces into smaller fragments, off-chain proof shard storage (blocks contain only outputs and proof references), and distributed verification where each proof shard is assigned to a validator subset through uniform deterministic assignment. An inference SPC opcode output is considered fully validated when all associated proof shards are validated by the quorum of assigned validators. This system is also compatible with broker-based solutions like Resonance.
            Lifecycle: The EOVMT lifecycle works in four steps: (1) A sole executor is selected via Resonance; (2) The executor processes the request and generates succinct sub-proofs; (3) Sub-proofs are gossiped and verified by validator subsets rather than re-executing the entire workload; (4) Validators assert correctness, and the rest of the network applies state changes without redundant re-execution.
            Flexible Proof Systems: Symphony supports multiple proof systems — TEE Attestations for rapid hardware-backed verification, Zero-Knowledge Proofs for stronger cryptographic and privacy guarantees, and Hybrid Solutions combining both for fast confirmation with added security.
            Safety and Liveness: Liveness is guaranteed through economic incentives similar to Ethereum 2.0's reward/penalty model. Safety is probabilistic — with at least ⅔ of total staked value controlled by honest participants and uniformly staked validators, the chance of a safety violation is negligible, comparable to a secure hash collision.
            Key Benefits: With EOVMT, consensus is optimized (single-node execution), decentralized (distributed proof verification), and streamlined (collective validity consensus). This eliminates inefficiencies of both replicated execution and replicated verification, while natively supporting node specialization.

    9. Guardians
        Guardians are Ritual Chain's advanced compute firewalls for specialized node operation, enabling node operators to granularly control which requests they service while still maintaining full consensus participation in the network.
        The Problem: Traditional blockchain nodes face a fundamental challenge — they must process all transactions uniformly, regardless of computational requirements or content. This creates an inefficient "one-size-fits-all" approach with three key limitations: Uniform Computation (nodes must process all transactions regardless of complexity, maintain expensive uniform hardware, and handle all computation types equally), Limited Specialization (current networks lack node specialization mechanisms, granular compute selection, and content-aware transaction processing), and Resource Inefficiency (leading to overprovisioned hardware, underutilized resources, and reduced operator diversity).
        Ritual's Solution: Just as traditional firewalls filter network traffic, Guardians enable computational and semantic filtering of blockchain operations. These filters empower node operators to granularly provision the exact set of requests they want to service while still participating in network consensus.
        Computational Firewalls operate on two dimensions. First, Hardware-based filtering allows nodes to filter based on AI model size, ZK circuit complexity, and TEE code execution requirements. Second, Resource-based filtering enables dynamic filtering by resource requirements including GPU memory thresholds, CPU core requirements, and network bandwidth needs.
        Statistical Firewalls enable nodes to hyperspecialize, only processing prompts and workloads that align with their intended use cases. For example, a node could choose to only process scientific computing tasks that fall within an embedding distance from core scientific topics. These operate on two dimensions: Semantic-based filtering (by prompt categories and topics, model specialization domains, and content safety boundaries) and Metric-based filtering (using embedding space distances, semantic similarity scores, and topic cluster membership).
        Consensus Preservation: Critically, while nodes can opt out of direct computation through Guardians, they still maintain network security by verifying succinct proofs from executor nodes, participating in Symphony's consensus, and contributing to network diversity — ensuring the network remains robust and decentralized even with highly specialized nodes.

        EIP Extensions: Ritual offers early support for in-demand EIPs including EIP-665 (Ed25519 signature verification), EIP-5027 (removal of smart contract code size limits with dynamic gas metering), EIP-5920 (a new PAY opcode to send ether without transferring execution context), and EIP-7212 (secp256r1/P-256 signature verification).
        Expressive Compute: Heterogeneous compute precompiles powering net-new applications, including Classical ML Inference, LLM Inference, ZK Proving & Verification, TEE Execution (privacy-preserving AI model inference, autonomous agents, external API calls), and Chain Abstraction (natively reading state from other blockchain networks).
        AI Primitives: Models are treated as enshrined, first-class blockchain objects with special-purpose primitives across provenance, privacy, and computational integrity — including Enshrined On-Chain AI Models, Verifiable Provenance, and a Model Marketplace for transparent monetization.
        Enshrined Oracles: Priority execution for critical automated operations like oracle updates, powered by Infernet ↔︎ World (enshrined oracles built into the chain) and Scheduled Transactions (top-of-block privileged inclusion).
        And Much More: Additional features include Modular Storage (supporting HuggingFace and Arweave), Resonance (surplus-maximizing transaction fee mechanism), and Ritual ↔︎ World (accessing Ritual's compute from any blockchain network).


RITUAL FOR NODE RUNNERS
    Currently, you can participate in running nodes on Ritual by operating an Infernet node. Ritual Chain itself is in private testnet, but the architecture is designed to enable broad participation when it goes public. No specific hardware is strictly required — whether you have a basic home setup or datacenter-grade hardware, Ritual's node specialization system will ensure your node contributes meaningfully. Nodes can specialize in AI Inference, ZK Proving, or TEE Code Execution, and features like Guardians (a granular firewall system) and Native Infernet Integration allow operators to balance security, performance, and compute delegation. More ways to participate will be announced as Ritual continues to open-source.
