What is the Ritual architecture?
    - EVM++
    EVM++ is defined as a backwards-compatible extension of the Ethereum Virtual Machine (EVM) designed to enable the creation of "actually smart" contracts
    It achieves this by integrating expressive compute precompiles, native scheduling, built-in account abstraction, and several high-demand Ethereum Improvement Proposals (EIPs)
    The key innovations of EVM++ include:
        - Native Scheduling: EVM++ introduces a "Scheduler" predeploy contract that allows developers to register specific callback frequencies for their smart contracts
            This enables recurring and conditional transaction execution without the need for external keepers
            Block producers manage these pending invocations and can include them at the top of blocks, with the Resonance mechanism ensuring timely allocation for compute-intensive tasks
        - Native Account Abstraction: By including EIP-7702 (Set EOA Account Code), EVM++ blurs the distinction between Externally Owned Accounts (EOAs) and smart contract accounts
            This allows users to replace standard EOAs with accounts that have programmatic functionality
            Expressive Compute and Execution Sidecars: At the core of EVM++ are heterogeneous compute precompiles that allow for net-new applications, such as AI inference, ZK proof verification, and TEE code execution
            These are powered by Execution Sidecars, which generate the necessary outputs and proofs consumed by the Ritual execution client while maintaining a standard precompile interface for developers
        - EIP Extensions: EVM++ prioritizes the inclusion of EIPs that improve developer and user experiences, which often lag on other chains
            Notable examples include EIP-665 (Ed25519 signature verification), EIP-5027 (removing smart contract code size limits), EIP-5920 (a PAY opcode for ether transfers), and EIP-7212 (secp256r1 signature verification)
        - Future Asynchronous Execution: Looking ahead, EVM++ aims to support long-running precompiles through native asynchronous execution, allowing users to perform computationally demanding work while still inheriting the security properties of the chain

    - Execution Sidecars
    Execution Sidecars are optimized components within the Ritual architecture designed to support expressive, heterogeneous compute—such as AI inference, zero-knowledge (ZK) proof verification, and Trusted Execution Environment (TEE) code execution
    They function as a specialized layer that allows the blockchain to handle complex tasks that go beyond standard smart contract capabilities
    These sidecars run and scale parallel to the Ritual execution client, invoked through abstraction via the EVM precompile interface. By default, these sidecars are purpose-built for execution and proof generation avoiding blockchain-specific overhead, run as an isolated process or remote to an execution client, and can be selectively enabled by node operators based on their available infrastructure, underlying hardware, or firewalled preferences.
    Because of their generic architecture, sidecars enable Ritual to easily service heterogeneous compute across any supported execution environment by connecting to their precompile-like interfaces (EVM precompiles, SVM precompiled programs, MoveVM system modules, etc.).
    Resonance is a state-of-the-art transaction fee mechanism designed to efficiently match supply and demand, providing users with optimally priced transaction execution

    - Resonance 
    Unlike traditional mechanisms like Ethereum's EIP-1559, which requires every node in a network to execute every transaction, Resonance operates as a two-sided fee market that matches specific sets of user transactions to specific capable, executing nodes
    The core concept relies on several key participants and mechanisms:
        1. The Two-Sided Market
            Resonance facilitates an exchange between two primary groups:
                Users: Instead of specifying gas fees (base and priority fees), users assign a valuation to their transactions, representing the maximum amount they are willing to pay for execution. This valuation remains encrypted and private to the network until a successful match is found
                Nodes: Resonance enables node specialization and selective execution. Each node specifies a cost function, which declares the true cost for that specific node to execute a particular bundle of transactions. Like user valuations, these cost functions are kept private to protect node margins.
        2. Surplus Maximization
            The primary goal of Resonance is surplus maximization. This is the process of achieving the highest total economic surplus in the market by setting the optimal price and quantity for transaction execution. By doing so, Resonance ensures its allocations are the most economically efficient, meaning no other matching could result in better prices for users or higher earnings for nodes.
        3. Key Entities
            Brokers: These are sophisticated, profit-seeking agents that compute efficient allocations by matching transactions to sets of nodes. They act similarly to "searchers" or "block builders" in the MEV ecosystem, pocketing the spread between a user's valuation and a node's execution cost.
            Auctioneer: This entity orchestrates the brokers and executes the Resonance mechanism. It consists of an on-chain smart contract (handling registration, payments, and disputes) and an off-chain service (which receives transactions, initiates batches, and selects the surplus-maximizing routing provided by brokers).
        4. Implementation and Lifecycle
            Resonance is an opt-in upgrade implemented through account abstraction. Users can choose to use Resonance for specific transactions by upgrading their wallets to a specialized reference implementation (AuctioneerWallet.sol), which allows them to use both Resonance and standard EIP-1559 transactions in parallel. The transaction lifecycle under Resonance involves the Auctioneer collecting and decrypting valuations and cost functions, finalizing a surplus-maximizing routing, and then having assigned nodes execute the transaction rather than the entire network

    - Symphony
    Symphony is a new consensus protocol designed by Ritual to reduce replicated execution and support high-performance, resource-intensive workloads, such as AI inference. In traditional blockchains like Ethereum, every node must execute every transaction to reach consensus; however, Symphony introduces a more efficient model to handle "heterogeneous" compute that requires significant resources or time.
    Ethereum transactions consist of a sequence of opcodes executed sequentially by the EVM. Today, these transactions (and thus, opcodes) are executed in a replicated fashion, where all execution clients, of all validators, execute all transaction opcodes to come consensus over execution state.
    The EVM is purposefully optimized for homogeneous, resource-constrained computation with strict resource limits (see: block gasLimit). This homogeneity is what affords Ethereum the ability to perform synchronous replicated execution across all nodes.
    Ritual enables heterogeneous compute to enrich what users can do on-chain. We enhance the EVM by introducing new operations that enable AI Inference, ZK Proving & Verification, TEE Execution, and more. Executing these operations, unlike all EVM opcodes today, requires heterogeneous resources that not only exceed the typical capabilities of validator nodes, but may also take upwards of minutes or even hours to execute.
    To support these heterogeneous operations, while preserving the guarantees offered by the Ethereum blockchain, Symphony introduces three key innovations:
    1. Execute-once semantics
        Symphony adopts an execute-once model for heterogeneous operations.
            Transaction operations are partitioned into two sets:
                - Homogeneous operations (existing opcodes)
                - Heterogeneous operations (unique Ritual functionality)
        Heterogeneous operations are executed only once by selected, staked executor nodes, who gossip their execution outputs to other validators.
        When transactions containing these operations appear in a block proposal, rather than re-executing the resource-intensive computation, validators directly slot in the gossiped execution outputs into their EVM state.
        To ensure correctness of these gossiped outputs without recomputation, Symphony employs succinct non-interactive proofs of execution, which are prover-optimized (making it much more efficient to assert correctness).
    2. Distributed verification via partitioned proofs
        While the execute-once model resolves scalability issues in execution, verifying proofs for heterogeneous operations remains challenging. Taking LLM Inference as an example:
            Proof sizes, though succinct, grow polylogarithmically with model complexity
            Single blocks may contain many transactions, each generating multiple proofs
            This inflates block sizes, hinders network propagation, and increases verification overhead
        To address these challenges, Symphony introduces three key innovations in distributed verification:
            Large models are divided into smaller sub-models, each with their own sub-proofs
            Blocks contain only outputs with off-chain references to associated proofs
            Validators are assigned sub-proofs deterministically, ensuring that only a subset of validators verify each sub-proof. The network accepts sub-proofs as valid when its assigned validators agree to quorum validity
        This approach improves propagation and verification efficiency, minimizes block sizes, and reduces the per-validator verification load while still upholding trust properties.
    3. Optimized committee election
        Optimized committee election allows Symphony to handle scenarios where no succinct proof system exists or where verifying such a system would be prohibitively expensive.
        In these situations, the protocol designates a smaller group of validators who work together to validate the operation outputs, forming a quorum on correctness. Quorum results can be adopted quickly—serving as a fast-path to justified execution—while still remaining compatible with Symphony’s broader principle of execute-once-verify-many-times.

    - Scheduled Transactions
    Scheduled transactions enable recurring, conditional invocation of smart contract functions, at the top of a block, without the need for external keepers.
    Homogenous compute
    We introduce a new predeploy contract, Scheduler. This minimal contract, callable only by other smart contracts (including EIP-7702 delegated addresses), maintains state of ScheduledCall(s) and escrows future transaction fees.
    Developers can easily set up scheduled smart contract invocations through Scheduler.schedule(), which takes as inputs details about an invocated function and calldata, invocation frequency, an optional conditional execution function (returning True/False), and fee limit preferences. Developers can cancel future invocations and receive remaining escrowed balance refund via Scheduler.cancel().
    Next, we extend the standard Ethereum block-building process. Block producers maintain the latest state of ScheduledCall(s) from the Scheduler. During intervals when a ScheduledCall should be executed, nodes include a SystemTx transaction (akin to L2 networks like Optimism) invoking a developer smart contract at the top of their produced block, conditional on transaction validity and sufficient fees. To developer smart contracts, this transaction acts as any other invocation, except with a msg.sender equal to a constant SystemTxAccount.
    ​   Heterogeneous compute
        Scheduled transactions containing heterogeneous compute consume the same, standard Scheduler interface, but are executed via the Resonance fee mechanism.
        Today, we hijack the mechanism by just-in-time pricing each invocation of a ScheduledCall as if it were an independent, unscheduled transaction. Brokers are given a one-block heads-up to upcoming scheduled transactions that they can include in future allocations. These transactions are executed by assigned subsets of the network with outputs and proofs gossiped to block producers for inclusion in similar fashion to the homogenous case.
    ​   Top-of-block inclusion
        Uniquely, scheduled transactions have privileged inclusion at top-of-block.
        This enables priority execution for critical, automated operations like oracle updates and rebalancing mechanisms, with separately priced fee markets.

What's new in Ritual?
1. EVM++
    EVM++ is a backwards-compatible extension of the Ethereum Virtual Machine with expressive compute precompiles, native scheduling, built-in account abstraction, and in-demand EIPs — enabling developers to build "actually smart" contracts while using familiar tooling.

    1. Overview
        EVM++ has four core components:
            - Scheduled Transactions — Recurring, conditional execution without external keepers
            - Account Abstraction — Native smart contract accounts via EIP-7702
            - EIP Extensions — Continual support for in-demand EIPs improving UX/DX
            - Expressive Compute — Heterogeneous compute precompiles (AI inference, ZK proving, TEE execution, etc.) powered by optimized execution sidecars
        The architecture is designed so developers can progressively adopt advanced Ritual Chain functionality. Behind the precompile interfaces are modular sidecars, and thanks to Resonance's fee mechanism, supporting new compute types or execution environments (SVM, MoveVM, etc.) is naturally supported. Looking forward, Ritual plans to enable long-running precompiles via native asynchronous execution.
    2. Account Abstraction
        Background: Ethereum has two account types — EOAs (controlled by private keys, limited functionality) and smart contract accounts (controlled by code, unlimited functionality). Smart contract accounts unlock batch calls, gas sponsorship, arbitrary signing keys (P256, BLS), and advanced ACLs.
            - EIP-7702 (SetCodeTx): Ritual Chain is one of the earliest blockchains to support EIP-7702, proposed by Vitalik Buterin. This new transaction type lets EOAs delegate a smart contract as their implementation, effectively giving EOAs programmatic smart contract functionality by signing and sending a transaction to `address(self)`.
            - Use Cases: Account abstraction powers Resonance (users upgrade wallets via `AuctioneerWallet.sol` to let the Auctioneer manage ETH balances) and Smart Agents (secure wallet management and delegation for autonomous on-chain agents).
    3. Scheduled Transactions
        The Problem: Developers need to periodically execute smart contract functions (parameter updates, protocol operations). Today, this requires off-chain keeper bots from third-party services like Gelato or Chainlink Automation — forcing reliance on external entities without execution guarantees.
        Native Scheduling: Ritual introduces a `Scheduler` predeploy contract. Developers call `Scheduler.schedule()` with function details, calldata, invocation frequency, optional conditional execution functions (returning True/False), and fee preferences. They can cancel via `Scheduler.cancel()` and receive refunds.
        How It Works:
            - Homogeneous compute: Block producers maintain pending `ScheduledCall` state and include `SystemTx` transactions at the top of each block, conditional on validity and sufficient fees
            - Heterogeneous compute: Uses the same `Scheduler` interface but executes via the Resonance fee mechanism, with brokers given one-block heads-up for upcoming scheduled transactions
        Key Benefits:
            - Top-of-block priority — Guaranteed priority execution for critical operations like oracle updates
            - Keeper-free automation — No external keeper networks needed
            - Conditional logic — Optional view functions dictate real-time invocation conditions
            - Dedicated fee markets — Separately priced from standard transactions
            - Optimized cost via Resonance
        Looking Forward: Resonance will natively price scheduled transactions within the mechanism itself, and true PBS++ (proposer-builder separation) will let brokers account for top-of-block inclusion during block construction.
    4. EIP Extensions
        The Problem: Ethereum and EVM-based chains have historically lagged in adopting novel EIPs due to development overhead or naive gas pricing for precompiles.
        Ritual's Approach: Via Resonance and Symphony, Ritual's architecture is purpose-built to price, orchestrate, and expose new EIPs before any other chain. Current extensions include:
            - EIP-665
            Adds an `ED25519VFY` precompile at `address(0x9)` for Ed25519 signature verification at 2000 gas. Ed25519 is used by TLS 1.3, Signal, Tor, and GitHub SSH. Takes 128 octets input (32-byte message, 32-byte public key, 64-byte signature) and returns 4 octets (0x00000000 for valid).
            - EIP-5027
            Removes the limit on smart contract code size with new, dynamic gas metering — enabling larger, more complex contracts.
            - EIP-5920
            Introduces a new `PAY` opcode (`0xf9`) that transfers ether without transferring execution context, eliminating reentrancy and DoS attack vectors that have historically led to billions in exploits. It pops `addr` and `val` from the stack and transfers `val` wei to `addr` without calling any functions.
            - EIP-7212

2. EVM++ Sidecars
    1. Overview
        EVM++ Sidecars are modular, containerized compute modules that run parallel to Ritual's execution client, exposed through the standard EVM precompile interface. Unlike traditional EVM precompiles implemented directly in execution client codebases, Ritual's compute precompiles are implemented as disjoint sidecars. They are purpose-built for execution and proof generation, run as isolated processes, and can be selectively enabled by node operators. This generic architecture lets anyone who can build a containerized sidecar automatically enshrine their computation on-chain. Sidecars are currently stateless but will soon become stateful for asynchronous, long-running compute.

        Supported sidecars: Classical ML Inference, LLM Inference, ZK Proving & Verification, TEE Execution, Network Calls, and Chain Abstraction.

    2. AI Inference
        Two sub-categories:
            - Classical ML Inference — Leverages canonical tree-based and regression-based models for efficient, scalable ML computation.
            - LLM Inference — Supports any model architecture, any inference engine (TGI, llama.cpp, ONNX), and any hardware profile (CPU, GPU, NPU, Apple Silicon). Ritual provides a universal inference layer that abstracts infrastructure complexity, verifiable execution via Symphony's dual proof sharding, and sovereign deployment with no central infrastructure dependencies. It also supports fine-tuning via the vTune framework.
    3. Network Calls
        The Network Call Sidecar enables smart contracts to initiate arbitrary HTTP requests as part of on-chain execution. It overcomes oracle limitations by supporting long-tail endpoints (REST, GraphQL, gRPC), permissioned APIs with encrypted secrets, and chaining multiple HTTP calls in a single atomic transaction. Requests execute inside a TEE for isolated, attestable, tamper-resistant execution with credential protection. Use cases include dynamic DeFi protocols accessing real-world data, unconstrained smart contracts calling off-chain AI providers, cross-platform integrations with web services, and on-chain agent infrastructure.

    4. ZK Proving & Verification
        This sidecar extends the EVM with native zero-knowledge proof generation and verification. It abstracts away cryptographic complexity, providing a standardized interface for generating and verifying proofs. It supports multiple proving systems and circuits, letting developers choose trade-offs between proof size, generation time, and verification cost. It's optimized for AI workloads (exploiting model symmetry for parallelized proof generation/verification) and addresses key limitations of current ZK systems: computational overhead, lack of AI-native operations, fragmented tooling, and trusted setup requirements.

    5. TEE Execution
        The TEE sidecar performs secure computation in hardware-enforced Trusted Execution Environments (secure enclaves). It enables privacy-preserving AI model inference, autonomous & self-evolving agents, and external API calls. TEEs provide near-native performance, real-time execution, data confidentiality, and tamper prevention. Trade-offs include trust in chip manufacturers and susceptibility to side-channel hardware attacks. TEE is one of Ritual's modular computational integrity gadgets alongside ZKML, OPML, and PPML.

    6. Chain Abstraction
        This sidecar enables smart contracts on Ritual to natively read (and in the future, write) state from other blockchain networks. Today, it integrates with Wormhole Guardian's attested query system for secure cross-chain data reads. Ritual Chain serves as a multi-chain coordination hub — a Schelling point for autonomous agents operating across multiple blockchains, providing a unified state view and synchronized operation layer. Use cases include cross-chain yield optimization, multi-chain arbitrage, cross-chain governance, and multi-agent coordination.

7. Symphony
    The Problem: Replicated Execution
        Most blockchains today operate on replicated execution — all nodes re-execute each block to:
            Verify transactions through re-execution
            Compute and apply state updates to their local state
        This model worked well for simple, homogeneous computations (basic arithmetic, hash functions). However, Ritual enables heterogeneous, resource-intensive workloads (AI inference, ZK proving, TEE execution) that not only exceed typical validator capabilities but may take minutes or even hours to execute. Replicated execution becomes redundant, inefficient, and unscalable.

    What Is Symphony?
        Symphony is a new consensus protocol that leverages dual proof sharding, distributed verification, and optimized committee election to reduce replicated execution and enable high-performance, resource-intensive workloads.

        While originally designed as an extension to Ethereum's Gasper, Symphony is consensus-gadget agnostic and can work with aBFT protocols, which Ritual initially targets.

    Core Model: Execute-Once-Verify-Many-Times (EOVMT)
        Symphony builds upon an EOVMT model where select compute nodes act as sole executors of workloads, generating succinct sub-proofs for executed compute.

        Three Key Innovations:
            1. Execute-Once Semantics
                Transaction operations are partitioned into two sets:
                    Homogeneous operations (existing EVM opcodes)
                    Heterogeneous operations (unique Ritual functionality — AI inference, ZK, TEE, etc.)
                Heterogeneous operations are executed only once by selected, staked executor nodes. These nodes gossip their execution outputs to other validators. When transactions containing these operations appear in a block proposal, validators directly slot in the gossiped outputs into their EVM state rather than re-executing.

                To ensure correctness without recomputation, Symphony employs succinct non-interactive proofs of execution that are prover-optimized. Since SPC opcode execution and proof generation occur asynchronously off-chain, proof generation does not impact the on-chain critical path.

                Ritual prioritizes prover-friendly systems with linear or near-linear proof generation time, further optimizable through parallel proof generation using multi-core or multi-machine setups.

            2. Distributed Verification via Partitioned Proofs
                While execute-once resolves execution scalability, verifying proofs remains challenging because:

                Proof sizes grow polylogarithmically with model complexity
                Single blocks may contain many transactions generating multiple proofs
                This inflates block sizes, hinders network propagation, and increases verification overhead
                Symphony introduces a four-pronged scalability strategy:

                Strategy	Description
                    Model Partitioning: Large models are divided into smaller sub-models, each with their own sub-proofs, producing smaller proofs that can be efficiently propagated and verified
                    Proof Sharding: Each sub-proof can be further divided into proof shards by breaking the computation trace into smaller fragments, leveraging the inherent structure of the proof system
                    Off-chain Proof Storage: Blocks contain only outputs with off-chain references to associated proofs, keeping verification metadata minimal
                    Distributed Verification: Sub-proofs are assigned to different validator subsets through uniform deterministic assignment. A quorum of assigned validators verifies each proof shard, and their consensus is accepted by the network
                An output is considered trusted and fully validated when all associated proof shards are validated by the quorum of assigned validators. EOVMT is also compatible with broker-based solutions for verifier assignment, such as Resonance.

            3. Optimized Committee Election
                For scenarios where no succinct proof system exists or verification would be prohibitively expensive, Symphony designates a smaller group of validators who work together to validate operation outputs, forming a quorum on correctness. Quorum results serve as a fast-path to justified execution while remaining compatible with EOVMT's broader execute-once-verify-many-times principle.
                Flexible Proof Systems
                    Symphony is proof-system agnostic, supporting multiple proof technologies:
                        TEE Attestations — Initial implementation uses Trusted Execution Environments for rapid verification with hardware-backed security guarantees
                        Zero-Knowledge Proofs — Parallel development offering stronger cryptographic guarantees, especially for privacy-sensitive computations
                        Hybrid Approaches — Future implementations may combine TEEs for fast transaction confirmation with ZK proofs for additional security assurances
                EOVMT Lifecycle
                Step-by-step:
                    Sole executor selected — A single node is selected via Resonance's optimized fee mechanism and matching protocol
                    Executor processes request — The node executes the compute workload, generating succinct sub-proofs in the process
                    Sub-proofs gossiped and verified — Generated sub-proofs are verified by a subset of validator nodes (not re-executed)
                    Validators assert correctness — The decentralized subset ensures all sub-proofs are valid; the rest of the network applies state changes without redundant re-execution or verification, relying on the safety guarantees of the consensus protocol
                Key Benefits
                    Optimized: A single node executes resource-intensive workloads and generates sub-proofs
                    Decentralized: Each sub-proof is assigned to a subset of validators in a decentralized manner
                    Streamlined: The network collectively reaches consensus on execution validity through validators sharing their approvals on sub-proofs' verification
                    Eliminates inefficiencies of both replicated execution (all nodes re-execute) and replicated verification (all nodes verify all proofs)
                    Introduces native support for node specialization
                Safety and Liveness
                    Liveness: Guaranteed through an incentive mechanism ensuring compute and validator nodes are economically motivated to perform their roles (similar to Ethereum 2.0's rewards and penalties)
                    Safety: Adopts a probabilistic (computational) safety approach. With at least ⅔ of total staked value controlled by honest participants and uniformly staked validators, Symphony leaves only a negligible chance of safety violation, comparable to the probability of a secure hash collision
                Background: Consensus Context
                Symphony is informed by two existing consensus approaches:
                    Ethereum 2.0 (Gasper): Hybrid PoS using Casper FFG for finalization and LMD-GHOST fork-choice rule; finality takes multiple epochs
                    CometBFT: Pure BFT protocol achieving single-shot finality when >⅔ of validators pre-commit; mostly asynchronous with weak synchrony assumptions for liveness
8. Node specialization
    Background:
        Traditional blockchains like Ethereum operate by coming to consensus over ordering and replicated execution of transactions. Step-by-step, it looks something like:
            - Block proposer elected: A leader, more commonly referred to as a block proposer, is elected to extend the chain by proposing a block (today, via stake-weighted pseudo-random selection). This block proposer collects transactions from both public and private transaction pools (using by running a sidecar like mev-boost), builds a valid block according to protocol rules, and broadcasts it to the rest of the network.
            - Validator re-execution: Next, all validators in the network receive the new head block from the block proposer and re-execute the block locally, via their execution clients, to determine block validity. Today, in Ethereum (via the official specification), this involves asserting:
                    - A block header is valid
                    - All transaction(s) are valid by their validity rules
                    - The sum of transaction gasLimit(s) do not exceed the block gasLimit
                    - A block’s stateRoot matches a local stateRoot after executing all transactions
                    - …and similar txsRoot, withdrawalsHash, ommersHash, and other checks
                Notice, in step 4, that Ethereum validators, irrespective of their hardware or specifications, must be capable of executing all Ethereum transactions.
            Achieving consensus: Finally, validators that have re-executed and asserted validity of a new head block publish their votes in favour of the block. Via the fork choice rule, all correct nodes eventually agree on a common view of the canonical chain, finalizing a block to Ethereum state.
        Limitations
            Today, the Ethereum Virtual Machine is purposefully limited in the types, computational complexity, and cost of operations it supports.
            In part, this is because of a conflict in Ethereum’s architecture demanding each node execute all transactions, yet prohibitively low minimum requirements to run a node, optimized for the long-tail of residential validators.
            Commonly, proposals to improve chain performance or enable net-new user functionality are met with pushback to preserve the minimum requirements.
                Namely, this results in:
                    Only supporting limited, homogeneous computation optimized for the weakest node
                    Best-case chain performance dependent on worst-case node performance
                    No user preference over compute execution; all nodes paid the same irrespective of hardware, capabilities, or performance
        Optimized Architecture
            Ritual Chain introduces node specialization through architecture purpose-built for reducing redundant re-execution and enabling user preference:
                Symphony reduces replicated execution
                    Symphony is a new consensus protocol leveraging dual proof sharding, distributed verification, and optimal sampling to reduce replicated execution.
                    At core, the principle behind Symphony is to execute-once-verify-many-times. Single nodes are selected for transaction execution (via Resonance). In addition to generating execution outputs, these nodes also generate succinct computation proofs.
                    In place of transaction re-execution by all validators, subsets of nodes now verify succinct proofs and broadcast transaction validity, with the network collectively reaching execution consensus.
                    Via Symphony, re-execution is made redundant, with nodes free to service just their specialized computation, while still participating in validating the network.
                Resonance enables user preference
                    Resonance is a new, state-of-the-art fee mechanism built for heterogeneous compute.
                        Traditional blockchains like Ethereum:
                            - Inefficiently price unique resources (computation, storage, etc.) as identical
                            - Force users to pay fees subsidizing transaction re-execution across all nodes
                            - Limit users from expressing their execution preferences
                        Resonance is a two-sided marketplace to match heterogeneous transactions to heterogeneous nodes.
                        Via Resonance, users can express their preferences to prioritize cost or speed, with transactions efficiently matched to specialized execution nodes.
        Benefits
            Via this architecture, Ritual Chain is the most expressive blockchain in existence, built to meet the demands of complex on-chain applications and enable net-new user functionality:
                Node specialization optimizes network performance, allowing for smoother, more tailored processing across different workloads.

        Why Expressive Blockchains?
            Traditional blockchain architectures were designed for homogeneous transactions—simple transfers and basic smart contract interactions where every node executes the same computation. However, the Web3 landscape is rapidly evolving:
                Complex Computations: Modern blockchain applications demand increasingly sophisticated operations, from ZK proofs to ML inference
                Specialized Hardware: Different types of computation benefit from specialized hardware (GPUs, TPUs, FPGAs, ASICs)
                Resource Intensity: Heavy computational tasks like proof generation or model inference are inefficient when replicated across all nodes
                Cost Efficiency: Forcing all nodes to execute all computations leads to higher network execution costs and reduced throughput
                This evolution demands a new paradigm: one where blockchain networks can efficiently handle heterogeneous computation while maintaining security and decentralization.
        Ritual Architecture
            Three key architectural innovations make Ritual the ideal venue for expressive blockchain computation:
                Resonance
                    Resonance introduces a surplus-maximizing transaction fee mechanism that efficiently matches user transactions with specialized compute nodes. Unlike traditional blockchain architectures where all nodes execute all transactions, Resonance enables:
                    Selective execution based on node specialization
                    Private, unique execution cost functions for compute providers
                    Efficient price discovery through sophisticated brokers
                    ​
                Symphony
                    Symphony is a new state-of-the-art consensus protocol that builds upon proven consensus mechanisms while introducing:
                    Dual proof sharding for reduced replicated execution
                    Distributed verification for high-performance workloads
                    Optimized committee election for specialized compute tasks
